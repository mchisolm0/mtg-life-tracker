# Mana Ledger MVP Sync Architecture

Status: Accepted for MVP

## Decision

Mana Ledger's MVP sync model is local-first for active gameplay and server-authoritative for shared state:

- MMKV is the durable local/offline store on each device.
- Convex is the remote source of truth for synced matches, ownership, and canonical event history.
- The UI updates optimistically immediately on every life tap, before network confirmation.
- Every life change carries a globally unique `clientEventId` generated by the device.
- MVP conflict policy: only the owning player/device can modify that player's score. Other devices may observe the score but cannot submit life changes for that player.

The app should treat Convex as canonical once a mutation is acknowledged or a realtime query snapshot is received, but active match controls must remain usable while offline. Local MMKV state is the user's working copy plus an outbox of pending events to flush when Convex is reachable.

## Goals

- Instant tap feedback for life total changes, with no network dependency in the critical UI path.
- Durable offline operation for an in-progress match.
- Idempotent event sync so retries cannot double-apply life changes.
- A conflict model simple enough for MVP: player/device ownership instead of multi-writer merge UI.
- A path to later add poison, commander damage, match history, tournament admin overrides, and richer conflict handling without changing the basic event/outbox shape.

## Non-goals for MVP

- Multi-device simultaneous editing of the same player's score.
- CRDT-based counters or peer-to-peer sync.
- Server-side match adjudication beyond idempotency, authorization, and ownership checks.
- Admin/judge overrides. Those should be explicit future event types with separate authorization.
- Replacing Convex's canonical state with local state after the match is synced.

## Terminology

- Local match: the MMKV copy of match state on a device.
- Canonical match: the Convex copy of match state derived from accepted events.
- Optimistic event: an event already applied to local UI but not yet acknowledged by Convex.
- Outbox: the local MMKV queue of events waiting for Convex submission or retry.
- Owning device: the device that is currently allowed to submit changes for a specific player slot.

## Data model

### IDs

Use opaque strings everywhere. Recommended MVP format:

- `deviceId`: generated once at install, persisted in MMKV, e.g. `dev_${uuid}`.
- `matchId`: generated by Convex for shared matches; local-only matches may use `local_${uuid}` until created remotely.
- `playerId`: generated when a player slot is created, stable for the match.
- `clientEventId`: generated for every local event before applying it, e.g. `${deviceId}:${monotonicLocalSequence}:${uuid}`.

`clientEventId` is the idempotency key. Convex must reject duplicate `clientEventId` values by returning the previously accepted result, not by applying the delta again.

### TypeScript types


## Event shape

MVP only needs `lifeChanged`. Future counter types should follow the same envelope.

```ts
type MatchEvent = {
  type: 'lifeChanged';
  clientEventId: string;
  matchId: string;
  playerId: string;
  ownerDeviceId: string;
  delta: number;
  previousLife: number;
  nextLife: number;
  localSequence: number;
  createdAt: number;
};
```

Required validations:

- `clientEventId`, `matchId`, `playerId`, and `ownerDeviceId` are non-empty strings.
- `delta` is a non-zero integer. MVP UI normally emits `-5`, `-1`, `+1`, or `+5`, but the backend can accept any reasonable integer within a bounded range such as `[-100, 100]`.
- `nextLife === previousLife + delta` for client integrity.
- The player exists in the match.
- The submitting device/user owns the player slot.
- The match is not ended.
- `clientEventId` has not already been accepted for a different payload.

Convex should store enough event data to rebuild current life totals if needed, but the canonical match document may keep denormalized current player totals for cheap realtime query rendering.

## MMKV local storage keys

Use separate keys so the app can update the hot path without rewriting unrelated data.

```ts
const keys = {
  deviceId: 'mana-ledger:deviceId',
  activeMatchId: 'mana-ledger:activeMatchId',
  match: (matchId: string) => `mana-ledger:match:${matchId}`,
  event: (clientEventId: string) => `mana-ledger:event:${clientEventId}`,
  outbox: 'mana-ledger:outbox',
  sequence: 'mana-ledger:localSequence',
};
```

Suggested stored values:

- `deviceId`: `DeviceId` string.
- `activeMatchId`: current `MatchId` or local match id.
- `match:<matchId>`: serialized `LocalMatch`.
- `event:<clientEventId>`: serialized `LocalQueuedEvent`.
- `outbox`: ordered `ClientEventId[]`, oldest first.
- `sequence`: monotonic number incremented before creating each event.

## Local queue states

`localOnly`
: Event belongs to a local-only match that has not yet been created in Convex. It may be displayed and persisted, but the sync worker must first create/link the remote match.

`pending`
: Event has been applied optimistically and is eligible for upload.

`syncing`
: Event is currently being submitted to Convex. This is an in-memory/transient state and should be reset to `pending` on app start if no request is actually in flight.

`acked`
: Convex accepted the event. Store `convexEventId`, `acknowledgedAt`, and remove it from the active outbox.

`retry`
: Submission failed with a transient network/server error. Keep the optimistic local result, increment `attempts`, and set `nextAttemptAt` using backoff.

`rejected`
: Convex rejected the event permanently, usually because ownership or match state validation failed. The app must reconcile to canonical Convex state and surface a non-blocking warning if the local visible total changed.

Allowed transitions:

```text
localOnly -> pending -> syncing -> acked
localOnly -> pending -> syncing -> retry -> syncing -> acked
localOnly -> pending -> syncing -> rejected
pending -> syncing -> rejected
syncing -> pending  // app restart / unknown request outcome
retry -> pending    // retry window elapsed
```

## Client write flow

1. User taps a life control.
2. Read the current local player life from MMKV-backed state.
3. Increment local sequence and generate `clientEventId`.
4. Build `LifeChangedEvent` with `previousLife`, `delta`, and `nextLife`.
5. Apply `nextLife` to local UI immediately.
6. Persist the updated `LocalMatch`, `LocalQueuedEvent(status: 'pending')`, and append `clientEventId` to `outbox`.
7. Trigger the sync worker if online.
8. If Convex acknowledges the mutation, mark the event `acked`, store server metadata, remove it from active outbox, and keep the UI as-is unless the realtime canonical snapshot differs.
9. If Convex rejects the event, mark it `rejected`, fetch/apply canonical match state, and show a small warning such as "Score change was not accepted because this device does not own that player."

The UI should never wait for step 8 before changing the displayed total.

Conceptual tables:

```ts
matches: {
```

Indexes:

- `matchEvents.by_clientEventId` unique/idempotency lookup.

- `matches.by_ownerDeviceId` only if needed for device-owned active matches.

## Convex mutation/query contracts

### `matches.createOrResume`

Input:

```ts
{
  localMatchId?: string;
  deviceId: string;
  startingLife: number;
  players: Array<{
    playerId: string;
    displayName: string;
    ownerDeviceId: string;
  }>;
}
```

Output:

```ts
{
  match: ConvexMatch;
  serverTime: number;
}
```

Behavior:

- Creates a canonical match if no existing remote match is linked.
- Initializes each player's life to `startingLife`.
- Sets MVP player ownership from the provided owner device ids.
- Returns the canonical match for local reconciliation.

### `matches.getActive`

Input:

```ts
{ matchId: string }
```

Output:

```ts
{
  match: ConvexMatch;
  recentEvents: ConvexLifeEvent[];
}
```

Behavior:

- Realtime query subscribed by active match screens.
- Returns canonical current totals and recent accepted events ordered by `serverSequence`.
- Client uses this snapshot to reconcile acknowledged/rejected local state.

### `matches.submitLifeEvent`

Input:

```ts
{
  matchId: string;
  event: LifeChangedEvent;
}
```

Output:

```ts
{
  accepted: true;
  eventId: ConvexEventId;
  clientEventId: ClientEventId;
  serverSequence: number;
  matchVersion: number;
  canonicalPlayerLife: number;
  canonicalMatch: ConvexMatch;
} | {
  accepted: false;
  clientEventId: ClientEventId;
  reason: 'notOwner' | 'matchEnded' | 'playerMissing' | 'invalidDelta' | 'duplicateClientEventId';
  canonicalMatch: ConvexMatch;
}
```

Behavior:

- First validates the event envelope, including that `event.matchId` matches the submitted `matchId`.
- Then checks `clientEventId`. If the event was already accepted with the same payload, return the existing accepted result without applying it again.
- Rejects with `duplicateClientEventId` if the same `clientEventId` exists with a different payload.
- Loads the match and player.
- Enforces the MVP ownership rule: `event.ownerDeviceId` and authenticated identity, when present, must match the player's owner fields.
- Applies `delta` to the canonical player life in a transaction, appends `matchEvents`, increments `serverSequence` and match `version`, and returns the updated match.

## Reconciliation rules

### Startup

- Load `deviceId`, active match, and outbox from MMKV.
- Reset any persisted/transient `syncing` events to `pending` because the previous request outcome is unknown.
- Render local state immediately.
- Subscribe to `matches.getActive` if `matchId` is remote and the device is online.
- Start flushing eligible `pending`/`retry` outbox events.

### Convex snapshot received

- Treat Convex as canonical for all acked events and all players this device does not own.
- For locally owned players, compute display state as: canonical Convex total plus pending local deltas that are not acked/rejected yet.
- Mark local queued events `acked` when their `clientEventId` appears in `recentEvents` or mutation output.
- If an event is missing from Convex and still pending/retry, keep it in the optimistic overlay.
- If Convex's player owner fields no longer match this device, disable controls for that player and keep observing only.

### Mutation acknowledgement

- If accepted, persist `acked` metadata and remove the event from the active outbox.
- If accepted but `canonicalPlayerLife` differs from the local optimistic result after applying still-pending events, correct local state to the canonical-plus-pending calculation.
- If rejected, persist `rejected`, remove from active outbox, and rebase the displayed match from `canonicalMatch` plus any remaining valid pending local events.

### Offline and retry

- Network failure does not roll back optimistic life changes.
- Retry with exponential backoff and jitter while the app is active.
- Retry immediately when the app regains connectivity or returns to foreground.
- Preserve outbox order per match to keep UI/debug reasoning simple. Do not submit two events for the same player concurrently in MVP.
- If a retry receives an idempotent duplicate accepted response, mark the event `acked`.

### Conflict policy

MVP has no multi-writer merge for the same score.

- A life change is valid only if submitted by the owning player/device for that `playerId`.
- Non-owning devices must render controls disabled or hidden for that player.
- If ownership validation fails, Convex rejects the event; the client marks it `rejected` and snaps back to canonical state for that player.
- Ownership transfer, judge override, and admin correction are future explicit mutations/events, not implicit conflict resolution.

## UI implications

- Tap latency target is local-state speed; no spinner on each tap.
- Pending sync may be shown with a subtle match-level indicator, not per-tap blocking UI.
- Disabled non-owned player panels should still show realtime totals.
- Rejections should be rare; when they happen, use a small toast/banner and reconcile automatically.
- The current large tap zones stay the source of optimistic events.

## Acceptance criteria

- Architecture docs state that MMKV is the local/offline store and Convex is the remote source of truth.
- Every life tap creates a `lifeChanged` event with a unique `clientEventId` before the UI is updated.
- The UI updates optimistically from local state with no required network round trip.
- Local storage includes active match state, individual queued events, and an ordered outbox.
- Queue states and allowed transitions are documented.
- Convex mutation contracts include idempotent duplicate handling by `clientEventId`.
- Convex authorization/conflict handling enforces that only the owning player/device can change that player's life total.
- Reconciliation rules define startup, snapshot, acknowledgement, rejection, and offline retry behavior.
- Rejected events are not silently ignored: local state is corrected to canonical-plus-valid-pending and a warning can be shown.
- Non-owning devices can observe but not modify player scores in MVP.
- The design does not require implementing Convex, MMKV, or UI code in this task.

## Future extensions

- Add event types for poison, commander damage, player joins/leaves, match end, and judge/admin overrides.
- Add ownership transfer with explicit consent or tournament admin authority.
- Add compacted snapshots for long event histories.
- Add signed-in user identity as the primary owner and keep `deviceId` as an offline fallback.
- Consider SQLite or a richer local database if match history queries outgrow MMKV key/value access.
